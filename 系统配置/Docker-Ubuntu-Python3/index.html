1. 背景

  • 首先，我希望自己的Mac系统更清爽、干净，我不希望因为某些因素，去频繁的折腾重装系统和软件这件事。 而且，因为我是个正在学习入门编程的普通用户，在学习过程中，很难避免因为学习和测试，而进行的不可预测结果的安装、卸载软件或类库的动作，这些动作至少在我个人内心中，感觉会对系统产生影响，为了避免哪怕是心理层面的忧虑，我也希望把系统的日常使用和学习环境做一个隔离。 以往，我都会把学习和测试环境放在虚拟机里进行，做错了、弄乱了直接还原或重装系统，但是因为虚拟机的容量、运行速度和运行时对系统带来的速度影响，还是希望能够有更轻量、便捷的解决方案。
  • 一年前就知道了Docker这个技术，但没有去学习，这次尝试学习使用Docker，更主要是为了通过学习——来提升学习的能力。
  • Docker的定义，引自Wikipedia > Docker是一个开放源代码软件项目，让应用程序布署在软件容器下的工作可以自动化进行，借此在Linux操作系统上，提供一个额外的软件抽象层，以及操作系统层虚拟化的自动管理机制[1]。
  • Docker的优点
    • 对于我这个小白来说，其实没有什么优点，使用Docker，更多的只是为了了解一些新奇的技术。
    • 但是，这不代表Docker没有优点，相反的，看之前Docker的火热程度，Docker对于生产环境，优点还是很多的。关于Docker的讨论，比较详细的可以看知乎的这个问题Docker 有什么优势？。ps，吐槽一下IMB Blog系统的站内搜索，垃圾的一塌糊涂，简直没法用。
    • 我个人简单入门的体验，会在后面的总结中再说。

2. 目的

  • 2.1 学习使用Docker。
  • 2.2 尝试通过Docker技术，打造一个：清爽的Mac环境 + Docker测试环境 + Docker数据库环境，重要数据还是存储在Mac环境内。 我预计这样结构的好处：
    • 轻量、速度快。Docker的container容量更小，运行比虚拟机速度快。
    • 复用性极高。基于Dockerfile方法，重复构造container更加方便，复用性极高。哪怕重新安装Mac系统，依靠Docker，也可能更快速的复现之前的测试、数据库环境。

3. Docker常用基本命令

参考资料： 3.1. 全面的Docker学习资料，很推荐一个IBM的博客每天5分钟玩转容器技术，总计150多篇小文章，写的很清晰、易懂。 3.2. docker常用管理命令 - docker ps -a #查看所有container - docker rm ID #按ID删除一个container - docker search KEYWORD #按关键词KEYWORD检索一个可用的image - docker pull NAME #按image的NAME，下拉（下载）一个image - docker push NAME #把image推送到docker store，这个需要提前注册一个docker store账号 - docker login #登陆docker store - docker rmi ID #删除一个本地image - dockerfile ['RUN', 'CMD', 'ENTRYPOINT']，用法及区别，可以参考 #使用dockerfile可以创建一个结构化的文档，然后使用docker build命令，可以根绝这个文档内的命令，生成一个符合自己需求的image - docker build -t TAG NAME . #按当前目录下dockerfile的定义，构建一个image，'TAG NAME'是生成后image的名字 - docker run ['-d', '-p', '-v', '-it'] - '-d'参数是让服务在后台持续运行 - '-p'参数是映射容器内的指定端口到本机的制定端口。例如，容器内Jupyter服务运行在端口8888，我们希望在本机通过端口8888访问这个服务，那么就需要加上参数-p 8888:8888 - '-v'参数是映射容器内指定路径到本机的制定路径，这样可以让docker成为一个纯粹的运行环境，而数据存储放在容器的外部。参数格式-v [host path]:[container path]，例如，-v $PWD/data:/home/user/data - '-it'交互式访问container，类似ssh访问远程主机，参数后需要给出container内bash的工作路径，例如-it /bin/bash。这里需要注意，除dockerfilef方式，这类访问container的方式，在容器内所做的一切修改，以及数据的增加和删除（container内的数据，非通过-v设定的container外部路径存储的数据），如果不通过docker commit提交打包成为一个新的image，那么在stop这个container之后，会被docker缓存起来，后期可以通过docker ps -a查看，但是这个临时运行的container不会有TAG，所以事实上，后期很难会准确找到希望保留的临时container，除非stop后，立即通过docker commit打包成新的image。也就是说，本次从run之后到stop之间，对容器内环境的调整、数据的增删，都是临时性的，只会被缓存为一个没有明确标示的container，后期清理container时，很容易被删除或难以被找到，这样会导致遗失当时有价值的对container的所有环境调整和内部数据操作。这点是跟操作虚拟机环境里运行的linux系统，或ssh访问的远程系统，最大的不同。 这是docker的优点，通过这种方式，最大成程度的分离了生产环境和数据处理。 - docker stop ID #停止一个container的运行，结合前面的介绍，需要特别注意，在容器内所做的修改，在stop后，这些修改都会跟随这个container被缓存为一个没有TAG标识只有ID的container，但是，后期查找很麻烦，而且很容易遗忘甚至被删除。 - docker exec #进入一个运行中的container 参数 | 说明 ---|--- -u | 指定进入的容器以哪个用户登陆，默认是root -e | 设置进入后可以使用的环境变量，这样动态指定比较灵活 -d | 以后台方式执行，这样，我们执行完这条命令，还可以干其他事情，写脚本最常用 -i | 以交互方式运行，是阻塞式的 -t | 分配一个伪终端，这个参数通常与-i参数一起使用，然后， 在后面跟上容器里的/bin/bash，这样就把我们带到容器里去了。 实例： docker exec -u root -it mysql /bin/bash 退出： - exit #同时终止container运行 - CTRL+P再使用CTRL+Q #退出容器要注意了，如果使用exit命令，退出容器的同时把容器也停止掉了。如果退出容器后还让容器继续运行，就得先使用CTRL+P再使用CTRL+Q快捷键，这样就是退出容器但不停止容器。 - docker commit ID TAG NAME - 'ID'参数，指container的ID - 'TAG NAME'参数，指创建的image的名字 以上，是极小部分Docker常用的命令及参数，足够对Docker的简单操作使用。更多docker命令和参数的使用方法，还需要Google。

4. 通过Dockerfile创建image

在Docker里，有一个Registry的概念，是指存储image的仓库，在这个仓库里，有很多官方或个人用户，push的官方软件或个性化的image，可以通过Docker官方的store网站，来查找各种用途的image。 每一个image页面，都会有一个Overview的介绍标签，详细介绍这个image的用途和用法，另外，还会有一个Dockerfile标签，里面有构建这个image的Dockerfile，可以用来学习如何高效率的编写Dockerfile，或者pull下来，自己使用docker build来自己构建image。 本次我学习Docker，构建的Python3.6学习环境（Pandas、爬虫库），就是通过修改'alessandroadamo/ubuntu-ds-python3'这个image的Dockerfile而来的。 - alessandroadamo/ubuntu-ds-python3网址

5. 过程中遇到的

  • 5.1. 错误提示：delaying package configuration, since apt-utils is not installed 错误，这是个非严重错误，不会引起build的中断，可以忽略。
  • 5.2. 严重错误，会引起build终端。安装pyspider爬虫库时，遇到了各种依赖性所产生的错误，因为当时忘了截图，因此只引用部分错误信息如下： ```
    • error: openssl/ssl.h: No such file or directory
    • pycurl error: command 'x86_64-linux-gnu-gcc' failed with exit status 1
    • E: Unable to locate package python3.6-dev ``` 解决方法： 在安装并更新Python3和Pip3之后，通过apt-get安装这些库：python-dev, libcurl4-openssl-dev, libxml2-dev, libxslt1-dev, zlib1g-dev, openssl, libssl-dev, build-essential, zlibc, libidn11-dev, libidn11
  • 5.3. 非严重错误，错误提示找不到了。原因某个库的依赖关系，只支持小于4.5.3版本的tornado，因此安装时需要特别声明tornado的版本tornado==4.5.3
